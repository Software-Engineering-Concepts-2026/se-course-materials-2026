# ประกาศโจทย์โครงงานประจำภาคการศึกษา

รายวิชา: **วิศวกรรมซอฟต์แวร์ (Software Engineering)**  
ระดับชั้น: **นิสิตชั้นปีที่ 3**

เอกสารฉบับนี้กำหนดกรอบและข้อกำหนดของโครงงานประจำภาคการศึกษา โดยมุ่งเน้นให้นิสิตได้ฝึกกระบวนการทางวิศวกรรมซอฟต์แวร์อย่างเป็นระบบ ตั้งแต่การวิเคราะห์ความต้องการ การออกแบบระบบ การพัฒนา การทดสอบ การควบคุมเวอร์ชัน การทำงานเป็นทีม และการส่งมอบซอฟต์แวร์ที่มีหลักฐานเชิงกระบวนการตรวจสอบได้ 

โครงงานทุกหัวข้อจะต้องให้ความสำคัญกับ **กระบวนการพัฒนา** มากกว่าความซับซ้อนของเทคโนโลยี โดยเน้นการทำงานแบบทีม การสร้าง artefacts ที่สอดคล้องกัน และการพัฒนาระบบซอฟต์แวร์ขนาดเล็กถึงขนาดกลางที่สามารถสาธิตการทำงานได้จริง 

## 1. วัตถุประสงค์ของโครงงาน

โครงงานนี้มีวัตถุประสงค์เพื่อให้นิสิตสามารถ:

- ประยุกต์ใช้แนวคิดของ Software Development Life Cycle (SDLC) กับปัญหาจริงหรือกึ่งจริงได้อย่างเป็นขั้นตอน 
- ฝึกการจัดทำข้อกำหนดความต้องการของซอฟต์แวร์ ทั้ง functional requirements และ non-functional requirements 
- ออกแบบระบบโดยใช้แบบจำลองที่เหมาะสม เช่น Use Case Diagram, Class Diagram, Sequence Diagram หรือ State Diagram 
- พัฒนาซอฟต์แวร์แบบทำงานร่วมกันโดยใช้ Git, branching strategy, pull request และ code review 
- สร้างและดำเนินการทดสอบอัตโนมัติสำหรับ business logic ที่สำคัญของระบบ 
- ติดตามความคืบหน้าและจัดการงานพัฒนาในลักษณะ incremental หรือ agile อย่างเป็นระบบ 

## 2. เกณฑ์ร่วมของทุกโครงงาน

ไม่ว่านิสิตจะเลือกธีมใด ทุกทีมจะต้องพัฒนาโครงงานภายใต้ข้อกำหนดขั้นต่ำเดียวกัน เพื่อให้การประเมินผลมีมาตรฐานร่วมกัน และเพื่อสะท้อนหลักปฏิบัติพื้นฐานของวิศวกรรมซอฟต์แวร์ 

### 2.1 คุณลักษณะขั้นต่ำของระบบ

ทุกระบบจะต้องมีองค์ประกอบอย่างน้อยดังต่อไปนี้:

- ระบบผู้ใช้หลายบทบาท (Multi-role system) อย่างน้อย 3 บทบาท
- ฟังก์ชันจัดการข้อมูลพื้นฐาน (CRUD) อย่างน้อย 2 กลุ่มข้อมูลสำคัญ
- ขั้นตอนการไหลของงานหรือการเปลี่ยนสถานะ (Workflow / State Transition) อย่างน้อย 1 กระบวนการ
- Business rules ที่สามารถระบุเงื่อนไขและเขียน unit test ได้อย่างชัดเจน
- ส่วนติดต่อผู้ใช้ที่สามารถสาธิตการทำงานของระบบตามกรณีใช้งานหลักได้

### 2.2 มาตรฐานกระบวนการพัฒนา (Definition of Done)

#### GitHub Management

ทุกทีมต้องใช้ GitHub หรือ GitLab ในการบริหารโครงงาน และต้องแตกงานในกระดานงานแบบ Kanban เป็นงานย่อยที่มีขอบเขตชัดเจน เช่น “สร้างหน้าจอ CRUD สำหรับบันทึกประวัติวัคซีนสัตว์” ไม่ใช่งานกว้างเกินไป เช่น “ทำระบบสัตว์” 

#### Git Workflow

สมาชิกแต่ละคนต้องพัฒนางานผ่าน feature branch ของตนเอง และรวมโค้ดเข้าสู่ branch หลักผ่าน Pull Request เท่านั้น โดย Pull Request ทุกครั้งต้องมีการตรวจทานโค้ดจากสมาชิกในทีมอย่างน้อย 1 คนก่อนอนุมัติ 

#### Automated Testing

ทุกทีมต้องเขียน unit tests หรือ automated tests อย่างน้อย 10 test cases สำหรับ business logic สำคัญของระบบ และต้องรันผ่านระบบอัตโนมัติ เช่น GitHub Actions หรือเครื่องมือที่เทียบเท่า 

#### Process Tracking

การประเมินผลจะพิจารณาหลักฐานของกระบวนการทำงานจาก commit history, issue history, pull requests, code reviews และการอัปเดตงานอย่างต่อเนื่อง หากพบการนำโค้ดจำนวนมากมารวมส่งในช่วงสั้นก่อนกำหนดส่ง โดยไม่มีร่องรอยการพัฒนาอย่างสม่ำเสมอ อาจไม่ได้รับคะแนนในส่วนของกระบวนการ 

### 2.3 เอกสารและ artefacts ที่ต้องมี

ทุกทีมต้องจัดทำ artefacts อย่างน้อยดังต่อไปนี้:

- Product vision หรือคำอธิบายแนวคิดของระบบ 
- รายการผู้มีส่วนได้ส่วนเสีย (stakeholders) และขอบเขตระบบ 
- Software Requirements Specification (SRS) ฉบับย่อ หรือเอกสารข้อกำหนดที่มีโครงสร้างชัดเจน 
- แบบจำลอง UML อย่างน้อย 3 ชนิดตามลักษณะของระบบ 
- Traceability matrix แบบง่าย เชื่อม requirement กับ implementation หรือ test cases 
- Test cases / test report
- เอกสารสรุปการออกแบบ การติดตั้ง และการใช้งานระบบ
- เอกสารสะท้อนผลการทำงานของทีม (retrospective)

## 3. ธีมโครงงานที่เปิดให้เลือก

นิสิตแต่ละทีมเลือกพัฒนาได้ 1 ธีมจากรายการต่อไปนี้ โดยสามารถปรับรายละเอียดเชิงปฏิบัติได้ตามความเหมาะสม แต่ต้องไม่ลดทอนองค์ประกอบหลักที่กำหนดไว้ในเอกสารนี้

## 4. ธีมที่ 1: GreenCampus
### ระบบรายงานและพิกัดถังขยะแยกประเภทอัจฉริยะ

### 4.1 บริบทของปัญหา

มหาวิทยาลัยมีนโยบายส่งเสริมการคัดแยกขยะ แต่ยังพบปัญหาถังขยะบางจุดเต็มเร็ว ถังขยะบางประเภทชำรุด และผู้ใช้งานจำนวนมากไม่ทราบตำแหน่งของถังขยะแยกประเภทเฉพาะ เช่น ถังขยะรีไซเคิล ขยะอิเล็กทรอนิกส์ หรือขยะอันตราย ระบบนี้มีเป้าหมายเพื่อให้ชุมชนในมหาวิทยาลัยสามารถร่วมแจ้งเหตุ ค้นหาจุดทิ้งขยะที่เหมาะสม และช่วยให้เจ้าหน้าที่ติดตามงานได้อย่างเป็นระบบ

### 4.2 บทบาทผู้ใช้

- **Reporter**: นิสิตหรือบุคลากรทั่วไป ค้นหาตำแหน่งถังขยะและแจ้งเหตุปัญหาที่พบ
- **Staff**: เจ้าหน้าที่ผู้รับผิดชอบพื้นที่ รับงานและอัปเดตสถานะการดำเนินการ
- **Admin**: ผู้ดูแลระบบ จัดการข้อมูลถังขยะ ตรวจสอบภาพรวมการดำเนินงาน และดูข้อมูลสถิติพื้นฐาน

### 4.3 ฟังก์ชันหลักขั้นต่ำ

- ระบบจัดการข้อมูลถังขยะ (CRUD) เช่น ประเภทถัง ตำแหน่ง โซน และสถานะเบื้องต้น
- ระบบแจ้งเหตุเกี่ยวกับถังขยะ เช่น เต็ม ชำรุด วางผิดจุด หรือไม่พร้อมใช้งาน
- ระบบ workflow ของตั๋วแจ้งเหตุ โดยต้องควบคุมการเปลี่ยนสถานะอย่างมีลำดับ เช่น `Reported -> In Progress -> Resolved`
- ระบบมอบหมายหรือติดตามงานสำหรับเจ้าหน้าที่ในพื้นที่รับผิดชอบ

### 4.4 ประเด็นเน้นย้ำทางวิศวกรรมซอฟต์แวร์

- ควรมี **State Diagram** สำหรับอธิบายสถานะของตั๋วแจ้งเหตุ 
- ควรมีการทดสอบสิทธิ์ของผู้ใช้ในการเปลี่ยนสถานะ เช่น ผู้ใช้ทั่วไปไม่สามารถปิดงานแทนเจ้าหน้าที่ได้
- ควรระบุ business rules ของการจัดการตั๋วอย่างชัดเจนในเอกสารข้อกำหนด 

### 4.5 ฟีเจอร์เสริม (ถ้ามีเวลา)

- การจัดลำดับความเร่งด่วนของงาน
- แดชบอร์ดสถิติของจุดที่มีการแจ้งเหตุบ่อย
- การแนบรูปภาพประกอบการแจ้งเหตุ

## 5. ธีมที่ 2: Pawsitive Campus
### ระบบบริหารจัดการประชากรและสวัสดิภาพสัตว์จรจัดในมหาวิทยาลัย

### 5.1 บริบทของปัญหา

การดูแลสัตว์จรจัดในมหาวิทยาลัยเกี่ยวข้องทั้งเรื่องความปลอดภัย สุขภาพสัตว์ การฉีดวัคซีน การทำหมัน และการแจ้งเหตุฉุกเฉิน ระบบนี้มีเป้าหมายเพื่อช่วยจัดทำทะเบียนสัตว์ ติดตามข้อมูลการดูแล และสนับสนุนการประสานงานระหว่างชุมชน อาสาสมัคร และเจ้าหน้าที่ที่เกี่ยวข้อง

### 5.2 บทบาทผู้ใช้

- **Community Member**: นิสิตหรืออาจารย์ ดูข้อมูลสัตว์ แจ้งเหตุ และยื่นความประสงค์ขอรับเลี้ยง
- **Caregiver**: ผู้ดูแลหรืออาสาสมัคร บันทึกข้อมูลสัตว์ การให้อาหาร หรือพิกัดที่พบล่าสุด
- **Admin / Vet**: ผู้ดูแลระบบหรือผู้รับผิดชอบด้านสุขภาพสัตว์ อัปเดตประวัติการแพทย์ อนุมัติข้อมูล และพิจารณาคำขอรับเลี้ยง

### 5.3 ฟังก์ชันหลักขั้นต่ำ

- ระบบทะเบียนประวัติสัตว์รายตัว (CRUD) เช่น ชื่อเรียก รูปพรรณ รูปภาพ จุดพบประจำ และสถานะเบื้องต้น
- ระบบบันทึกประวัติทางการแพทย์ เช่น วัคซีน การทำหมัน หรือหมายเหตุด้านสุขภาพ
- ระบบคำขอรับเลี้ยงที่มี workflow ชัดเจน เช่น `Applied -> Interviewed -> Approved / Rejected`
- ระบบแจ้งเหตุเกี่ยวกับสัตว์ เช่น บาดเจ็บ ป่วย หรือแสดงพฤติกรรมก้าวร้าว

### 5.4 ประเด็นเน้นย้ำทางวิศวกรรมซอฟต์แวร์

- ควรมี **Class Diagram** ที่แสดงความสัมพันธ์ระหว่าง `Animal`, `MedicalRecord`, `Caregiver` และ `AdoptionRequest` 
- ควรมีการกำหนด business rules ชัดเจน เช่น สัตว์ที่มีสถานะรับเลี้ยงแล้วต้องไม่แสดงในรายการที่เปิดรับซ้ำ
- ควรมี unit tests สำหรับตรรกะสำคัญ เช่น การแจ้งเตือนสัตว์ที่ครบกำหนดวัคซีน หรือการเปลี่ยนสถานะคำขอรับเลี้ยง

### 5.5 ฟีเจอร์เสริม (ถ้ามีเวลา)

- เส้นเวลาแสดงประวัติการรักษา
- ระบบนัดหมายเพื่อรับเลี้ยง
- แผนที่จุดที่พบสัตว์ล่าสุด

## 6. ธีมที่ 3: Campus Share
### แพลตฟอร์มตลาดแบ่งปันและส่งต่อสิ่งของเหลือใช้ในมหาวิทยาลัย

### 6.1 บริบทของปัญหา

เมื่อสิ้นสุดภาคการศึกษา นิสิตจำนวนมากมีหนังสือเรียน อุปกรณ์หอพัก หรือสิ่งของที่ยังใช้งานได้แต่ไม่จำเป็นต้องใช้ต่อ หากไม่มีระบบกลางรองรับ สิ่งของเหล่านี้อาจกลายเป็นขยะโดยไม่จำเป็น ระบบนี้มีเป้าหมายเพื่อสนับสนุนการส่งต่อสิ่งของ การจองรับของ และการดูแลความน่าเชื่อถือของผู้ใช้งานในชุมชนมหาวิทยาลัย

### 6.2 บทบาทผู้ใช้

- **Giver / Seller**: ผู้ส่งต่อหรือผู้ประกาศรายการสิ่งของ
- **Receiver / Buyer**: ผู้ค้นหา จอง และรับสิ่งของ
- **Moderator**: ผู้ตรวจสอบรายการโพสต์ ดูแลปัญหาการใช้งาน และจัดการกรณีถูกรายงาน

### 6.3 ฟังก์ชันหลักขั้นต่ำ

- ระบบจัดการรายการสิ่งของ (CRUD) พร้อมหมวดหมู่ รายละเอียด และเงื่อนไขการรับของ
- ระบบจองสิ่งของที่มีสถานะอย่างน้อย เช่น `Available -> Reserved -> Completed / Cancelled`
- ระบบจัดการโพสต์ที่ถูกรายงานหรือไม่เหมาะสม
- ระบบให้คะแนนหรือข้อเสนอแนะหลังการส่งมอบเสร็จสิ้น

### 6.4 ประเด็นเน้นย้ำทางวิศวกรรมซอฟต์แวร์

- ควรมี **Sequence Diagram** สำหรับขั้นตอนการจองสิ่งของและการเปลี่ยนสถานะ 
- ควรกำหนด business rules ของการจองให้ชัด เช่น รายการที่ถูกจองแล้วไม่ควรถูกจองซ้ำ
- ควรมี unit tests สำหรับตรรกะของการจอง การยกเลิก หรือการคัดกรองเงื่อนไขที่ผิดกติกา

### 6.5 ฟีเจอร์เสริม (ถ้ามีเวลา)

- การยกเลิกการจองอัตโนมัติเมื่อเกินเวลาที่กำหนด
- การคัดกรองคำต้องห้ามในข้อความโพสต์
- ระบบจัดอันดับความน่าเชื่อถือของผู้ใช้

## 7. ระดับความคาดหวังของโครงงาน

เพื่อให้ทุกทีมสามารถบริหารขอบเขตงานได้เหมาะสม โครงงานควรถูกวางแผนเป็น 3 ระดับ ดังนี้:

| ระดับ | คำอธิบาย |
|---|---|
| **Minimum Requirements** | ฟีเจอร์หลักที่ทุกทีมต้องมี และเป็นฐานสำหรับการประเมินผลขั้นต่ำ |
| **Recommended Features** | ฟีเจอร์ที่ช่วยให้ระบบสมบูรณ์ขึ้นและสะท้อนความเข้าใจเชิงออกแบบได้ดีขึ้น |
| **Advanced Features** | ฟีเจอร์เสริมที่เพิ่มความท้าทายทางเทคนิค เช่น การจัดลำดับความสำคัญอัตโนมัติ การหมดเวลาอัตโนมัติ หรือการแจ้งเตือนอัจฉริยะ |

นิสิตควรเริ่มต้นจากการทำ Minimum Requirements ให้สมบูรณ์ก่อน จากนั้นจึงค่อยเพิ่ม Recommended หรือ Advanced Features ตามเวลาที่เหลือและศักยภาพของทีม 

## 8. ข้อกำหนดด้านการออกแบบและการทดสอบ

ทุกทีมต้องออกแบบระบบโดยให้ artefacts ต่าง ๆ มีความสอดคล้องกัน เช่น ความต้องการต้องเชื่อมโยงไปยัง use cases, โมเดล, งานพัฒนา และ test cases ได้อย่างตรวจสอบย้อนกลับ 

ระบบทดสอบต้องสะท้อน business logic สำคัญของระบบ ไม่ใช่เพียงทดสอบการแสดงผลพื้นฐานเท่านั้น ตัวอย่างเช่น การอนุญาตหรือปฏิเสธการเปลี่ยนสถานะ การจำกัดสิทธิ์ตามบทบาท การห้ามข้อมูลซ้ำ หรือการห้ามทำรายการที่ผิดลำดับของ workflow 

## 9. สิ่งที่อาจารย์จะใช้พิจารณาในการประเมิน

การประเมินผลโครงงานจะพิจารณาทั้ง **คุณภาพของระบบ** และ **คุณภาพของกระบวนการพัฒนา** โดยให้ความสำคัญกับประเด็นต่อไปนี้:

- ความครบถ้วนและความชัดเจนของ requirements
- ความสอดคล้องระหว่าง requirement, design, code และ tests
- คุณภาพของการออกแบบ workflow และ business rules
- การทำงานร่วมกันผ่าน version control อย่างเป็นระบบ
- คุณภาพของ commit history, pull requests และ code reviews
- ความครอบคลุมของ automated tests
- ความสามารถในการสาธิตระบบและอธิบายเหตุผลเชิงออกแบบ
- การสะท้อนผลการทำงานของทีมอย่างมีวิจารณญาณ

## 10. ข้อเสนอแนะในการเลือกหัวข้อ

นิสิตควรเลือกธีมที่ทีมสามารถเข้าใจบริบทของปัญหาได้จริง และสามารถเข้าถึงตัวอย่างผู้ใช้หรือสถานการณ์ใช้งานได้ง่าย เพราะจะช่วยให้กระบวนการเก็บ requirements และการออกแบบ use cases มีคุณภาพมากขึ้น 

การตัดสินใจเลือกธีมไม่ควรพิจารณาจาก “ความซับซ้อนของเทคโนโลยี” เป็นหลัก แต่ควรพิจารณาจากความสามารถของทีมในการทำให้ requirement, design, implementation, test, และ process evidence เดินไปด้วยกันอย่างครบถ้วน 

## 11. หมายเหตุสำคัญ

- อาจารย์ผู้สอนอาจอนุญาตให้ทีมปรับขอบเขตของฟีเจอร์บางส่วนได้ หากยังคงโครงสร้างหลักของระบบไว้ครบถ้วน
- ไม่อนุญาตให้เลือกหัวข้อใหม่ที่อยู่นอกเหนือจาก 3 ธีมนี้ เว้นแต่ได้รับอนุมัติล่วงหน้า
- ทุกทีมต้องสามารถอธิบายได้ว่าระบบของตนตอบโจทย์ใด ใครคือผู้ใช้หลัก และ business rules ใดคือหัวใจของระบบ
- ทุกทีมต้องพร้อมสาธิตระบบพร้อมหลักฐานของกระบวนการพัฒนาประกอบในวันนำเสนอ

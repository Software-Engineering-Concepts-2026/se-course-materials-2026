# 🏛️ แนวทางโครงการประจำภาคการศึกษา (Term Project Themes)
**รายวิชา:** วิศวกรรมซอฟต์แวร์ (Software Engineering) ชั้นปีที่ 3  
**เกณฑ์หลัก:** เน้นกระบวนการทางวิศวกรรม (Multi-role CRUD, Workflow State Machine, Automated Testing, Collaborative Git Workflow)

---

## 📌 คำแนะนำทั่วไปสำหรับทุกโครงการ (Definition of Done - DoD)
ไม่ว่านักศึกษาจะเลือกธีมใด ทุกโครงการจะต้องปฏิบัติตามมาตรฐานวิศวกรรมซอฟต์แวร์ดังต่อไปนี้:
1. **GitHub Management:** แตกตั๋วงานบน GitHub Projects (Kanban Board) อย่างละเอียดเป็นหน่วยย่อย (Granular Tasks) เช่น *"สร้างหน้าจอ CRUD สำหรับอัปเดตประวัติวัคซีนสัตว์"* ไม่ใช้ตั๋วงานกว้าง ๆ เช่น *"ทำหน้าบ้าน"*
2. **Git Workflow:** แยกทำงานรายบุคคลผ่าน Git Feature Branches และรวมโค้ดเข้าสู่ `main` branch ผ่าน **Pull Request (PR)** โดยต้องมีสมาชิกในทีมอย่างน้อย 1 คนทำ Code Review และกด Approved
3. **Automated Testing:** ออกแบบและเขียนคำสั่งทดสอบ (Unit Test Scripts) บน GitHub Actions อย่างน้อยทีมละ 10 Test Cases เพื่อควบคุม Business Logic ที่สำคัญของระบบ
4. **Process Tracking (วินัยใน Workflow):** คะแนนส่วนใหญ่จะวัดจากร่องรอยการเดินทางของโค้ด (Commit History) ที่อัปเดตอย่างต่อเนื่องสม่ำเสมอ การนำโค้ดจำนวนมากมาแปะรวมกันในคืนก่อนส่งจะส่งผลให้คะแนนกระบวนการเป็น 0

## 🚛 แนวทางที่ 1: ระบบรายงานและพิกัดถังขยะแยกประเภทอัจฉริยะ (GreenCampus)
### 1. บริบทและปัญหา (Problem State)
มหาวิทยาลัยมีนโยบายรณรงค์การแยกขยะ แต่ประสบปัญหาถังขยะบางประเภทเต็มเร็วเกินไป ถังขยะชำรุด หรือลูกบ้าน/นิสิตไม่ทราบว่าถังขยะรีไซเคิลแต่ละประเภท (เช่น ขยะอิเล็กทรอนิกส์ ขยะพลาสติก) ตั้งอยู่จุดใดบ้าง ทำให้เกิดการทิ้งขยะรวมกันจนเสียเปล่า แพลตฟอร์มนี้จะช่วยให้ชุมชนร่วมแจ้งเหตุและช่วยให้เจ้าหน้าที่เข้าจัดการได้อย่างเป็นระบบ

### 2. สิทธิ์ของผู้ใช้ (Roles)
* **Reporter (นิสิต/บุคลากรทั่วไป):** ค้นหาพิกัดถังขยะที่ใกล้ที่สุดแยกตามประเภท และเปิดตั๋วแจ้งเหตุถังขยะเต็ม/ชำรุด
* **Staff (แม่บ้าน/เจ้าหน้าที่จัดการสิ่งแวดล้อม):** รับตั๋วแจ้งเตือนในโซนที่รับผิดชอบ และอัปเดตสถานะการเข้าจัดเก็บ/แก้ไข
* **Admin (ผู้ดูแลระบบกลาง):** จัดการข้อมูลพิกัดถังขยะ (CRUD), ตรวจสอบประสิทธิภาพการทำงานของ Staff และดูแดชบอร์ดสถิติปริมาณขยะ

### 3. ฟังก์ชันหลักที่ต้องพัฒนา (Core Features)
* **Bin Registry (CRUD):** ระบบจัดการพิกัดถังขยะในแผนที่จำลอง พร้อมระบุประเภทขยะที่รองรับและความจุเริ่มต้น
* **Waste Ticket Reporting:** ฟังก์ชันให้นิสิตปักหมุดถังขยะที่เกิดปัญหา แนบภาพถ่าย และเลือกประเภทปัญหา (เช่น เต็ม, ชำรุด, วางผิดจุด)
* **Ticket State Machine (Workflow):** ระบบคุมสถานะของตั๋วแจ้งเหตุซึ่งต้องเปลี่ยนตามลำดับเท่านั้น: `Reported` ➡️ `In Progress` ➡️ `Resolved` (การก้ามข้ามสถานะอย่างผิดเงื่อนไขต้องถูกปฏิเสธ)
* **Priority Job Dispatch:** อัลกอริทึมหรือเงื่อนไขการจัดลำดับความเร่งด่วนของถังขยะ (เช่น ถังขยะติดเชื้อ/อันตรายที่เต็ม จะต้องถูกดันขึ้นบนสุดของบอร์ดงาน Staff)

### 4. ประเด็นเน้นย้ำทางเทคนิค (SE Focus)
* **UML & Spec:** การออกแบบ **State Diagram** เพื่อควบคุมสถานะของตั๋วแจ้งเหตุ
* **Unit Testing เคสตัวอย่าง:** ทดสอบว่าฟังก์ชันปิดตั๋วงาน (`Resolved`) จะต้องทำได้โดย Staff ผู้รับผิดชอบหรือ Admin เท่านั้น, ผู้ใช้ทั่วไป (Reporter) จะมาสั่งปิดตั๋วงานลอย ๆ ไม่ได้

## 🐕 แนวทางที่ 2: ระบบบริหารจัดการประชากรและสวัสดิภาพสัตว์จรจัด (Pawsitive Campus)
### 1. บริบทและปัญหา (Problem State)
ปัญหาสุนัขและแมวจรจัดในสถาบันมักสร้างความกังวลด้านความปลอดภัย การส่งเสียงรบกวน หรือการแพร่ระบาดของโรคติดต่อ (เช่น โรคพิษสุนัขบ้า) แต่การจัดการด้วยความรุนแรงไม่ใช่แนวทางที่ยั่งยืน แพลตฟอร์มนี้จะช่วยทำทะเบียนประวัติ ติดตามการฉีดวัคซีน การทำหมัน เพื่อควบคุมประชากร และเปิดโอกาสให้คนในชุมชนมีส่วนร่วมดูแลหรือรับเลี้ยงตัวที่ผ่านการฝึกพฤติกรรมแล้ว

### 2. สิทธิ์ของผู้ใช้ (Roles)
* **Community Member (นิสิต/อาจารย์):** ส่องดูทำเนียบสัตว์ สังเกตพิกัดอัปเดตล่าสุด แจ้งเหตุสัตว์บาดเจ็บ/ก้าวร้าว หรือยื่นความจำนงขอรับเลี้ยง (Adopt)
* **Caregiver (ชมรมคนรักสัตว์/อาสาสมัคร):** ขึ้นทะเบียนประวัติสัตว์ บันทึกรอบการให้อาหาร และอัปเดตพิกัดล่าสุดที่พบ
* **Admin / Vet (สัตวแพทย์/เจ้าหน้าที่มหาลัย):** อัปเดตประวัติการแพทย์ (วัคซีน, ทำหมัน), อนุมัติการขึ้นทะเบียนสัตว์ และพิจารณาคำขอรับเลี้ยง

### 3. ฟังก์ชันหลักที่ต้องพัฒนา (Core Features)
* **Animal Profile Management (CRUD):** ระบบทำทะเบียนประวัติสัตว์จรจัดรายตัว (ชื่อเรียก, รหัส, รูปพรรณ, พิกัดประจำ, รูปภาพ)
* **Medical Tracking Timeline:** ระบบบันทึกประวัติการแพทย์แบบเรียงลำดับเวลา (เช่น วันที่ฉีดวัคซีนล่าสุด, สถานะการทำหมัน `Yes/No`)
* **Stray Adoption Workflow:** ระบบยื่นคำขอรับเลี้ยงและการพิจารณาตามขั้นตอนสถานะ: `Applied` ➡️ `Interviewed` ➡️ `Approved/Rejected`
* **Incident Reporting & Alert:** ฟังก์ชันแจ้งเหตุฉุกเฉินเกี่ยวกับสัตว์ (เช่น สัตว์แสดงอาการดุร้าย หรือบาดเจ็บ) ระบบจะสร้างตั๋วงานส่งให้สัตวแพทย์หรืออาสาสมัครเข้าตรวจสอบทันที

### 4. ประเด็นเน้นย้ำทางเทคนิค (SE Focus)
* **UML & Spec:** การออกแบบ **Class Diagram** ที่ซับซ้อนปานกลาง แสดงความสัมพันธ์ระหว่างคลาส `Animal`, `MedicalRecord`, `Caregiver` และ `AdoptionRequest`
* **Unit Testing เคสตัวอย่าง:** ทดสอบ Business Logic การแจ้งเตือนเมื่อสัตว์ครบกำหนดรอบฉีดวัคซีนประจำปี หรือทดสอบเงื่อนไขว่าสัตว์ที่ติดสถานะ `Adopted` (มีคนรับเลี้ยงแล้ว) จะต้องไม่ถูกนำไปแสดงในรายการสัตว์ที่เปิดรับเลี้ยงซ้ำ

## 📚 แนวทางที่ 3: แพลตฟอร์มตลาดแบ่งปันและส่งต่อสิ่งของเหลือใช้ (Campus Share)
### 1. บริบทและปัญหา (Problem State)
เมื่อสิ้นสุดแต่ละภาคการศึกษา นิสิตชั้นปีต่าง ๆ มักจะมีหนังสือเรียน ชีทสรุป หรืออุปกรณ์หอพัก (เช่น พัดลม, โต๊ะญี่ปุ่น, ตู้เย็นเล็ก) ที่ไม่ได้ใช้งานแล้ว แต่ยังอยู่ในสภาพดี การทิ้งไปจะกลายเป็นขยะสถาบันและสร้างภาระให้ระบบจัดการสิ่งแวดล้อม แพลตฟอร์มนี้เป็นพื้นที่ส่วนกลางให้นิสิตนำสิ่งของมาโพสต์ส่งต่อให้รุ่นน้องฟรี (Free Sharing) หรือส่งต่อในราคาแบ่งปัน เพื่อลดปริมาณขยะและช่วยประหยัดค่าครองชีพ

### 2. สิทธิ์ของผู้ใช้ (Roles)
* **Giver / Seller (ผู้ส่งต่อ):** โพสต์รายการสิ่งของ รายละเอียด สภาพ และกำหนดเงื่อนไขการรับของ (เช่น นัดเจอที่โรงอาหาร หรือมารับเองที่หอพัก)
* **Receiver / Buyer (ผู้ขอรับ):** ค้นหาสิ่งของแยกตามหมวดหมู่ กดจองสิ่งของ และอัปเดตสถานะเมื่อได้รับของเรียจริง
* **Moderator (กรรมการนิสิต/เจ้าหน้าที่สโมสร):** ตรวจสอบและสกรีนโพสต์ที่ผิดกฎระเบียบ จัดการกรณีผู้ใช้ถูกรายงานว่ามีพฤติกรรมทุจริต หรือจองแล้วไม่มาตามนัด

### 3. ฟังก์ชันหลักที่ต้องพัฒนา (Core Features)
* **Item Listing Catalog (CRUD):** ระบบจัดการข้อมูลโพสต์สิ่งของ แยกตามหมวดหมู่ พร้อมระบุสภาพและเงื่อนไขการส่งต่อ
* **Time-bound Reservation (Concurrency Control):** เมื่อ Receiver กดจอง ระบบจะล็อกรายการนั้นและเปลี่ยนสถานะเป็น `Reserved` โดยให้เวลา 48 ชั่วโมงในการนัดรับ หากหมดเวลา ระบบต้องดีดสถานะกลับเป็น `Available` อัตโนมัติ
* **Trust & Rating System:** ระบบให้คะแนนความน่าเชื่อถือ (Peer Review Rating) หลังจากการส่งมอบเสร็จสิ้น เพื่อคัดกรองและป้องกันมิจฉาชีพ
* **Forbidden Content Filtering:** ฟังก์ชันตรวจจับและคัดกรองคำค้นหา (Keyword Linter) เบื้องต้น เพื่อปฏิเสธการโพสต์สิ่งของอันตรายหรือผิดกฎระเบียบของมหาวิทยาลัย

### 4. ประเด็นเน้นย้ำทางเทคนิค (SE Focus)
* **UML & Spec:** การเขียน **Sequence Diagram** เพื่อจำลองพฤติกรรมในขั้นตอนการจองสิ่งของและการตัดสิทธิ์เมื่อหมดเวลา
* **Unit Testing เคสตัวอย่าง:** ทดสอบระบบ Concurrency เบื้องต้น เช่น หากมีนิสิต 2 คนกดจองหนังสือเล่มเดียวกันในเวลาเดียวกัน ระบบต้องยอมรับ Pull คำขอแรกและตอบปฏิเสธ (Reject) คำขอที่สองโดยที่ข้อมูลไม่พัง (Data Consistency)
